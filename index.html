What Does It Mean to be OpenAI-Compatible?

  • An API mimics OpenAI's endpoints, request, and response formats.
  • Developers can use the same SDKs, CLI tools, and integrations without rewriting code.
  • Supports multi-provider usage and smooth switching.

Why Compatibility Matters

  • Portability: Easily switch AI providers.
  • Integration: VS Code, LangChain, CLI scripts rely on OpenAI-compatible APIs.
  • Standardization: Shared response formats prevent bugs.
  • Future-proofing: Avoid vendor lock-in.

Let's look at some examples:

1. Chat Completions (/v1/chat/completions)

Used for chat-based AI interactions.

{
  "model": "gpt-4",
  "messages": [
    {"role": "system", "content": "You are a helpful assistant."},
    {"role": "user", "content": "Write a haiku about coding."}
  ]
}
{
  "id": "chatcmpl-123",
  "object": "chat.completion",
  "choices": [
    {
      "index": 0,
      "message": {"role": "assistant", "content": "Code flows like a stream,\nLogic dancing in silence,\nDreams in every loop."},
      "finish_reason": "stop"
    }
  ],
  "usage": {"prompt_tokens": 15, "completion_tokens": 20, "total_tokens": 35}
}

2. Embeddings (/v1/embeddings)

Used for vector representations of text.

{
  "model": "text-embedding-ada-002",
  "input": "AI compatibility is powerful."
}
{
  "object": "embedding",
  "data": [
    {"index": 0, "embedding": [0.0123, -0.0045, 0.9987, ...]}
  ],
  "model": "text-embedding-ada-002",
  "usage": {"prompt_tokens": 6, "total_tokens": 6}
}

Response Message Format

Consistency is key: all providers through LiteLLM return OpenAI-compatible JSON.

{
  "id": "chatcmpl-abc",
  "object": "chat.completion",
  "choices": [
    {"message": {"role": "assistant", "content": "Hello world!"}}
  ]
}

Prevents downstream errors in clients expecting OpenAI schema.

Takeaway
